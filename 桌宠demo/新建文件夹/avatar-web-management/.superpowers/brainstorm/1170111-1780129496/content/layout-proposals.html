三种布局优化方案对比

基于 react-admin 布局模式，结合当前 Ant Design 技术栈

A

增量优化方案

保持现有结构，聚焦安全/a11y/响应式/性能的补齐

B

React-Admin 风格重构

采用 AppBar+Menu 分离、数据驱动CRUD、主题引擎等核心模式

C

混合渐进方案（推荐）

保留 Ant Design 组件体系，选择性引入 react-admin 最佳实践

方案 A：增量优化（最小改动）

当前结构 + 安全/a11y/响应式/性能补齐
✅ 改动最小仅修复已知问题，不改变用户体验
✅ 低风险无架构变更，回归测试范围可控
✅ 工期短约 3-5 天完成全部 P0-P2
❌ 不系统补丁式改进，后续维护成本可能递增
❌ 无模式提升不引入 react-admin 的数据驱动模式
🏗 改动范围：
1. CSP 头审计 + 输入消毒
2. aria 属性补全 + 焦点管理
3. 响应式断点统一
4. 图片懒加载 + 代码分割
5. 测试覆盖率提升

方案 C：混合渐进方案（⭐ 推荐）

核心原则：保留 Ant Design 组件，引入 react-admin 模式精华
保留不动
  • Ant Design 组件体系（Layout/Menu/Table/Card）
  • Next.js App Router 路由结构
  • Zustand 状态管理 + SWR 数据获取
  • Sidebar + Header + Content 布局结构
  • Tailwind CSS + Design Tokens 主题变量
借鉴 react-admin
  • List/Edit/Create/Show 统一页面模式（作为约定而非组件）
  • 筛选器 + 排序器组件抽象（``）
  • 批量操作工具栏（BulkActionBar）
  • 主题切换引擎（浅色/深色/系统）
  • 通知 + 撤销模式（乐观更新的回滚UI）
  • 仪表盘小部件系统
  • 焦点管理 + 键盘导航规范

推荐方案 C 核心改动清单

优点

  • 低风险：不改变现有架构，仅增强模式
  • 可逆：每个改动独立，可单独回滚
  • 实用：精准借鉴 react-admin 最有价值的模式
  • 工期适中：约 5-7 天分阶段交付
  • 不破坏：非管理后台功能（市场、社区、桌宠）不受影响

代价

  • 不如方案 B 系统化
  • 需要刻意维护模式一致性
  • 部分模式需自行实现（非开箱即用）
